# An Opus 4.8 test: Workflows, 9x9 chess with shared movement boards, and prompt construction romps

I decided to give out the new [Dynamic Workflows](TODO LINK TO POST) feature a test in Opus 4.8 as well as see how far I could push it with a 1-3 shot prompt.

The first attempt, I used Claude's mobile app (not Claude Code) to make the game as an artifact in pure chat.
This went shockingly well.
I was able to get a pretty complex game created with almost no issues, and immediately play it in under 45 minutes and 3 prompts.

To start things off, we will take a quick look at the original chat there, and then dive into how I pushed this further with Claude Code.

## The Original Chat

My original prompt to the chat was this:

We're going to make a chess game with multiple boards.
This will be complicated.

> It will be a 3x3 grid.
> The rules are simple enough.
> Standard chess with one addition: if your piece has an "ancestor" (a captured piece of the same type on another board) then if you are able to you can move it to another board.
> The goal is to get as many checkmates as possible.
> Build this logically.
> Use a functional core and an imperative shell.
> Types are essential - no any types and no string hacks.
> Everything should be as pure as possible.
> Everything should be unit tested and also tested compositionally.
> This will ensure you can one shot the implementation.

> You should be able to drag around the boards and zoom in and out to scope what you can do.
> Two fingers to pan, pinch to zoom on mobile.

> Use a simple modern design with tailwind or something to make it all pretty. This game should be single player for now, with the player playing both sides. Eventually we will allow myltiplayer but as we have no server this is a demo to begin things and build out the core game logic and premise.

> It should be packageable as a single page bundle and shareable as an artifact.

It did a really great job with 2 follow up prompts, eventually getting all the way to having a working bot that used real rudimentary strategy to play against.

## A try using this with "Dynamic Workflows"

Dynamic Workflows are a new Claude Code feature.
We will assume you read the post linked above in the introduction.

I decided to start things off by refining the prompt a bit to tailor it to working with some web development that was a bit more complex.
I assume I will add multiplayer here eventually, or maybe even get it to work as a React Native app or something.
Who knows, depends on how much time I have.

Here was the first prompt for this rendition:

> We are going to do a challenge.

> I want you to make a game.

> The game rules are as follows:

> 1. it is a 3x3 set of chess boards
> 2. all games are active at all times
> 3. standard rules for moves EXCEPT: if a piece of the same kind as the moving one has been captured (in a 1:1 relationship, if 1 bishop goes you cant bring over two) on the board you want to move to from your position, you may make the move
> 4. the game is played to the most checkmates possible
> 5. it should be responsive and work on desktop and mobile

> the game should be built with strict standards

> 1. typescript
> 2. functional core, imperative shell
> 3. no usage of Any() types, no string hacks, be truly strong contracts that represent the nature of the game(s)
> 4. unit testing all over with vitest
> 5. test the composition and possible game states for all edge cases

> for the other player, you are to implement a basic bot to play against. It cannot just make random moves.
> Get it to an easy to medium skill level

> it should be packageable as a single file that we can pass around and load and if dropped into claude code desktop, could be an artifact instantly

> do you have any questions? I would like you to utilize a workflow for this to test out your new capabilities

Now, this is like, a pretty complex-ish game?
Even if you got the rules right, it really complicates things when you get to moving pieces cross board and having 9 winning game states but only 1 possible move per turn etc.

I figured it would be something that most reasonable people would consider "a huge pain in the ass to write"

## Let's see what it does with this `workflow` keyword thing

It asked some clarifying questions that I think had obvious answers.
We can gloss over them for now.

The Dynamic Workflow started up after answering these.

Let's take a look at what it gave us:

```
⏺ Perfect — that settles it: a real, fully-tested strict-TS project, with a bundling step bolted on that
  emits one self-contained .html. That's the right engineering call.

  I'll lock the rest with the sensible defaults I proposed (you didn't push back on them):

  - Continuous 24×24 plane — board borders are invisible to geometry; sliding pieces glide across them.
  - One boundary crossing per move — keeps the move tree and rules sane.
  - Knights/pawns cross too — a knight's L-landing or a pawn's diagonal capture onto the neighboring board
  counts, gated by the same own-piece-captured credit.
  - Kings are board-bound — no king is ever captured, so no credit exists; per-board checkmate stays
  well-defined.
  - Checkmated board is frozen & scored — no pieces enter or leave; most boards mated wins.
  - Capture credit = your own captured pieces, 1:1.

  Let me do quick environment recon, then I'll kick off a design workflow to pin the type model before I
  implement.
```

So this is pretty clear, its getting started and understands what we want.

The word `workflow` shows up in a rainbow when you have the feature enabled and available, that is the tell.
From here, it kicked off a 3 subagents.

It had a flow.

> Now the design workflow.
> Three architects independently propose the type model + architecture; a judge synthesizes one blueprint. I'll embed the full spec.

> ⏺ Design workflow is running in the background. While it works, I'll scaffold the toolchain (independent of the design output) — strict TS config, vitest, ESLint with a hard ban on any, and the single-file bundler. Let me set these up.

Now, we see this checklist


So this is pretty legit.
It really is just cranking.
